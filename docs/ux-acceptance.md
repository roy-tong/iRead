# iRead 用户体验验收

本文档从非技术 Codex 用户的视角验收 iRead。功能测试通过不等于用户体验达标。

## 核心任务

用户只应该需要完成三个决策：

1. 输入一个或多个关注领域。
2. 检查并修改候选信源。
3. 选择报告深度，批准是否开始采集和定时运行。

用户不应该需要提供 RSS 地址、编写 JSON、查找配置目录、理解运行状态码，或自己判断下一条 CLI 命令。

## 测试矩阵

| 阶段 | 通过标准 |
| --- | --- |
| 安装 | 一条命令完成；缺少依赖时在任何副作用前说明修复方法；成功后给出可直接使用的下一句话 |
| 空环境 | 不将仓库示例领域或信源显示为用户订阅 |
| 领域理解 | 只询问会实质改变信源的地域、语言或用途区分 |
| 信源提案 | 五类信源角色齐全；每个信源有直接主页、2-3 篇代表作、冲突/风险和保守的冷启动评分 |
| 审批 | 给出可直接回复的批准、删除和暂停示例；静默绝不视为批准 |
| 激活 | 长任务持续报告阶段、已完成项和下一个门槛；中断后可从已保存状态继续 |
| 交付 | 最近一个自然月的回补达标；至少一份本地报告存在且可打开；所有覆盖缺口显式披露 |
| 跨任务 | 新 Codex 任务能自动找回订阅、最新报告和可恢复的下一步 |

## 性能目标

- 安装器应在 10 秒内给出首个可见状态。
- 领域研究应在 10 秒内说明当前阶段，每完成一个领域立即展示进度。
- 不得在 Codex 主任务中启动第二个 Codex 进程来生成信源提案。
- 本地提案校验应在 1 秒内完成。
- 任何超过 30 秒的可交互步骤都必须提供进度或持久化恢复点。

## 稳定版门槛

只有当一个全新的多领域订阅完成以下所有项目，才能发布 `v0.2.0`：

- 严格信源提案通过，且用户完成复核。
- 微信扫码模式或明确批准的 RSS/网页源模式完成激活。
- 过去一个自然月数据通过完整性审计。
- 定时任务是当前活动配置，且重启后仍可运行。
- 至少一份日报、一份周报和一份月报生成并通过人工阅读验收。
- `iread acceptance` 返回 `accepted: true`，或只有用户明确接受的非必需警告。

## Beta 4 实测结论

- 安装、空环境隔离、审批边界和跨任务发现已通过自动测试。
- 当前任务直接生成提案的路径已取代嵌套 Codex 进程。旧路径在实测中 220 秒仍无进度，不再作为 Codex 默认路径。
- 真实三领域订阅已采集 324 篇，但尚有 224 篇待分析、9 个必抓网页源待接入、0 份报告，且不是当前活动定时配置。
- 因此 Beta 4 适合继续公开测试，但尚未达到稳定版交付标准。
