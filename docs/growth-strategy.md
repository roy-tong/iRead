# iRead 开源增长策略

## 目标

iRead 的增长不是让更多人收藏一个仓库，而是让目标用户完成四个动作：

1. 在搜索、文章或他人分享中发现 iRead。
2. 在一分钟内理解价值，并完成安装。
3. 在一次任务中拿到值得审核的信源组合。
4. 在七天内持续收到有用的报告，并愿意推荐或贡献信源。

北极星指标是“每周产生至少一份合格报告的活跃订阅数”。Star 和页面浏览量只是上游信号。

## 增长漏斗

| 阶段 | 用户问题 | 当前动作 | 核心指标 |
| --- | --- | --- | --- |
| 发现 | 这是什么，和 RSS 有何不同 | GitHub 描述、Topics、中英文 README、社交预览、真实案例 | 搜索曝光、仓库独立访客 |
| 激活 | 我能否快速看到价值 | 一行安装、一句领域提示、批准前预览 | 安装成功率、首次提案时间 |
| 价值 | 推荐信源是否值得信任 | 五类信源组合、代表作、风险和覆盖缺口 | 提案批准率、信源保留率 |
| 留存 | 报告是否持续节省时间 | 日周月分工、质量规则、失败披露 | 七日活跃订阅、报告打开与反馈 |
| 传播 | 我为什么愿意推荐 | 可复现案例、可分享的元数据、领域贡献入口 | Star、Fork、领域需求、信源贡献 |

不加入默认遥测。安装和本地阅读数据不上传。早期量化依靠 GitHub Traffic、Issue、Release 下载和用户自愿反馈；若未来增加遥测，必须默认关闭并单独征得同意。

## 仓库发现优化

- 仓库描述同时包含 `AI research assistant`、`source discovery`、`RSS` 和 `daily/weekly/monthly digest`。
- Topics 覆盖用户搜索意图，不使用过细或重复词：`ai-research-assistant`、`rss-reader`、`source-discovery`、`research-tools`、`news-aggregator`、`daily-digest`、`self-hosted`、`codex`、`wechat`。
- README 首屏先给结果、安装命令和测试提示，再解释方法和边界。
- 每个 Release 使用同一结构：谁应升级、三项关键变化、安装命令、已知限制、完整 Changelog。
- 分享 GitHub 链接时使用 1280 x 640 社交预览图，保证项目名称和价值主张可读。

## 内容与渠道

第一阶段只做与真实使用强相关的内容，避免泛泛介绍“AI 阅读助手”。

| 内容 | 证据 | 适合渠道 |
| --- | --- | --- |
| 从三个领域到 27 个信源的完整过程 | 候选、淘汰原因、采集覆盖和时间 | GitHub、知乎、少数派、V2EX |
| 为什么“官方源越多”不等于研究质量高 | 五类信源角色与反例 | 公众号、即刻、行业社区 |
| 11 秒回补 318 篇后，日报如何去重 | 基准记录和报告质量规则 | Hacker News、Reddit、开发者社区 |
| 一个领域的公开信源包 | 代表作、风险、采集方式 | 对应行业社群与研究社区 |

每篇内容只设置一个行动：运行一行安装命令，或提交一个领域需求。不要同时要求 Star、关注公众号、加群和填写表单。

## 社区增长机制

1. **领域需求**：普通用户只写自然语言领域，维护者返回可复用的信源方案。
2. **信源纠错**：允许提交失效链接、归属变化、利益冲突和更好的代表作。
3. **公开案例**：只发布链接、元数据和原创分析；不得提交第三方全文、Cookie 或凭据。
4. **版本节奏**：Beta 期间每两周最多一个面向用户的版本，避免频繁升级打断测试。
5. **贡献回报**：Release Notes 明确感谢领域方案和信源贡献者，并链接对应 Issue。

## 30 天执行计划

### 第 1 周：把首次成功做稳定

- 找 5 位非开发者，只给 GitHub 首页，不口头解释。
- 记录安装完成时间、首次提案时间、困惑点和放弃位置。
- 修复所有导致用户无法在 10 分钟内看到信源提案的问题。

### 第 2 周：建立三个跨行业案例

- 选择监管、技术和消费三个不同领域。
- 每个案例披露候选数量、角色覆盖、自动采集比例、淘汰原因和首份日报。
- 由领域用户审核代表作和关键遗漏，不能只由维护者自评。

### 第 3 周：定向发布

- 发布一篇中文案例和一篇英文技术复盘。
- 在最相关的两个社区首发并回答评论，不同时铺十个平台。
- 所有链接附同一版本号和同一安装命令，减少支持分叉。

### 第 4 周：复盘并形成循环

- 查看 GitHub Insights 的 Traffic、Clones、Popular content 和 Referring sites。
- 汇总 Issue 中最常见的领域、失败步骤和信源纠错。
- 只优化漏斗中损失最大的一个环节，再发布下一版本。

## 发布前检查表

- README 中的安装命令从公共网络可用。
- 新机器安装在 30 秒内完成，失败时能给出单一恢复动作。
- 示例提示可以触发信源预览，且批准前不会采集。
- 至少一个公开案例与当前版本一致。
- Release Notes 明确 Beta 边界和不支持项。
- GitHub 描述、Topics、社交预览和版本号一致。
- 没有第三方全文、密钥、Cookie、本机私有路径或授权二维码进入提交。
