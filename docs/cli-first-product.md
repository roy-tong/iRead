# 第一阶段产品形态：CLI + Agent 接入

本文只说明第一阶段的技术交付形态。目标用户、完整用户旅程、领域和信源策略、采集验收、报告逻辑及产品路线以 `product-plan.md` 为准。

## 结论

第一阶段发布的是一个完整的CLI产品，能支持用户通过codex和workbuddy进行安装、调度、使用。

- `iread` CLI 是稳定产品接口，负责提案、配置、采集、分析、报告和导出。
- JSON 配置、Schema、Prompt 和信源策略都在仓库中公开、可审计、可版本化。
- Codex 插件把 CLI 包装成自然语言流程，适合没有代码基础的用户。
- WorkBuddy 适配器提供原生工作流 DAG 和逐项批准门槛。
- 文章资料库网页只负责阅读，不负责系统配置。

这种形态保留了开源项目的可组合性，也让普通用户通过 Agent 完成操作。后续 GUI 只需要调用同一组 CLI 和 Schema，不需要重做领域模型。

## 系统分层

```text
Codex Skill / WorkBuddy Workflow / Shell
                  |
                  v
            iRead CLI
                  |
       +----------+-----------+
       |          |           |
   Proposal    Collection   Reporting
   + review    + archive    + export
       |          |           |
       +------ JSON/SQLite ----+
```

完整代码仓是发布单元，CLI 是所有入口共享的执行面。Codex 和 WorkBuddy 只编排命令，不复制业务逻辑。

## 普通用户流程

### 1. 选行业或研究领域

用户只需给出一个或多个领域关键词，例如“医疗器械监管、新能源电力市场、数字人文”。语言、地区、受众和研究目标都有保守默认值；只有会显著改变选源结果时，Agent 才追问。示例只用于说明输入形式，产品没有领域白名单。

系统把输入扩展为：

- 3–8 个互斥度较高、可用于报告聚合的一级主题；
- 行业事件类型和关键词；
- 公司、机构和关键作者冷启动种子；
- 已知排除项和覆盖缺口。

### 2. 检查信源

候选信源按功能角色选取，而不是只堆“知名媒体”：

- `primary_source`：监管机构、公司、实验室、标准组织等一手材料；
- `expert_voice`：目标作者、研究者和一线从业者；
- `independent_reporting`：可用于交叉验证的独立报道；
- `specialist_analysis`：行业数据库、垂直研究和专业分析；
- `discovery_signal`：只负责发现线索，不直接作为高置信结论依据。

每个候选源必须展示主页、采集方式、覆盖主题、推荐理由、利益冲突、风险提示和 2–3 篇代表作品。用户只需要检查这张清单及代表作品，不需要自己寻找 RSS URL。

### 3. 选择报告策略

第一阶段提供三种统一预设：

- `light`：低阅读负担，日报聚焦重大变化，周月报保留趋势摘要；
- `standard`：平衡覆盖与阅读时间，适合作为默认值；
- `deep`：更多候选文章、事件和证据，适合专业研究。

每种预设独立定义日报、周报、月报是否启用、阅读时间、最大条目和关注重点。高级用户可以继续修改 `reporting.json`。

## 信源发现与评级

信源发现采用统一漏斗：

```text
领域关键词
  -> 主题、实体和事件扩展
  -> 按角色搜索候选源
  -> 验证主页、Feed 和代表作品
  -> 冷启动评分与冲突披露
  -> 用户批准
  -> 采集样本
  -> 用真实文章质量重新评级
```

冷启动评分包含领域匹配、权威性、原创性、证据规范和可采集性，并带置信度。它只是先验值。采集后，系统基于相关文章数量、文章相关度、原创性、证据质量、标题党风险、正文完整性和更新状态进行收缩估计；样本不足时不会给出虚假的高置信评级。完整算法见 `source-quality.md`。

## 批量与审批边界

批量清单可以包含多个领域。`batch-propose` 为每个领域生成独立 JSON，支持失败隔离和断点续跑。生成提案不会触发订阅、抓取或发布。

`apply-subscription` 必须收到明确批准的 domain id。Codex Skill 和 WorkBuddy 工作流都要求先展示信源及代表作，再收集批准项，并为整个订阅确认一个 `light`、`standard` 或 `deep` 策略。清单中的预设值只是建议默认值，不代表用户已经选择。`--approve-all` 只能用于用户明确批准全部领域的情况。

批准后的多个领域会合并进一个订阅目录。领域成为报告一级主题，原提案中的一级主题成为订阅内二级主题；共享信源按微信 ID、Feed、主页或名称去重，并记录其覆盖的全部领域。一个订阅共用数据库、采集连接和报告节奏，另一套独立订阅仍使用不同的 `runtime.json` 路径。

## 开源内容边界

仓库公开代码、Prompt、Schema、信源清单、评分策略、文章元数据和派生分析。第三方文章正文、图片、站点 HTML、付费内容和逐字稿不自动获得 MIT 授权。

因此公开归档默认只发布链接、元数据和结构化分析。只有来源明确允许转载或操作者拥有授权时，才能使用 `--include-content --rights-confirmed` 导出全文。这一限制不能被 Agent 自动绕过。

## 第一阶段验收标准

- 用户给出一个或多个行业关键词即可生成结构化研究提案。
- 每个提案包含角色平衡的信源、直接 URL、风险信息和代表作品。
- 用户可以按领域批准，并选择轻量、标准或深度报告策略。
- 多个已批准领域生成一个可被 CLI 加载的订阅目录，并可用 `iread subscription` 检查。
- Codex 与 WorkBuddy 调用同一 CLI，且不会绕过审批门槛。
- 开源导出默认不包含第三方全文。

GUI、托管服务、账号系统和面向终端用户的自动更新器不属于第一阶段范围。
