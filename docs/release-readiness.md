# iRead 发版就绪度

## 当前结论

当前版本定位为 `0.2.0-beta.5`。它已适合通过 GitHub 供 Codex 和 Claude Code 用户安装和反馈，但还不应标记为稳定版。豆包专业版和 WorkBuddy 暂列为实验性适配。

非技术用户任务、性能目标和稳定版验收矩阵见 [`ux-acceptance.md`](ux-acceptance.md)。

## 已具备

- 任意单领域或多领域订阅模型，不包含领域白名单。
- Codex 批量信源检索、提案生成、严格校验和人工批准边界。
- WorkBuddy 原生提案流程，不依赖另一台机器安装 Codex。
- Claude Code 个人 Skill、开放 Agent Skill 包和豆包可导入 ZIP；主 Skill 渐进加载并且不扫描源码仓库。
- 四个 Agent 入口的安装都由脚本执行，Doctor 默认只输出一行摘要，日常启动只读取一次 workspace。
- 多领域主题合并、共用信源去重、独立运行目录和统一报告策略。
- 日报、周报、月报生成，以及 RSS、微信公众号、Notion 和公开归档能力。
- MIT 许可证、第三方组件与信源披露、全文导出权利确认开关。
- 用户批准后的本地微信二维码授权、信源匹配、最近一个自然月回补、就绪验收和定时任务。
- 无公众号后台权限时的 RSS/网页源降级模式，以及不含第三方全文的定期本地公开归档。
- 必抓 `web_pending` 信源未连接时使用 `active_with_gaps` 并产生完整性告警，不把候选源误报为已采集。
- Codex 可以跨任务发现已登记订阅，继续授权或回补，检查定时任务和覆盖缺口，并定位最新的本地日报、周报和月报。
- Agent 可以通过 `capabilities` 读取权限和副作用契约；所有状态变更支持请求 ID 幂等和本地操作审计。
- `acceptance` 对配置、上下文、激活、采集、分析、调度、报告和覆盖质量进行统一结果验收。
- 生成订阅默认只输出本地报告；外部发布保持显式，定时任务可经批准卸载且不删除本地数据。
- 用户对报告、信源和订阅的反馈会保存在本地并进入后续报告的编辑偏好上下文。
- 公开 GitHub 仓库、Beta 标签、Release 和 GitHub Actions 已建立；运行数据、密钥和第三方正文未进入仓库。

## 稳定版前仍需完成

- 在一台干净电脑上完成 GitHub 克隆、Codex 插件安装、真实多领域配置和跨任务恢复验收。
- 使用新建订阅完成一次真实微信扫码路径，并验证不含微信源的 RSS/网页模式。
- 验收 Codex 中“打开最新报告”和“为什么没有出报告”的完整用户体验。
- 在新 Codex 任务中验收重复请求跳过、失败操作恢复、结果验收、反馈生效和停止定时任务的完整流程。
- 如需让 GitHub 自动公开本地归档，还需由仓库所有者明确开启并配置有限的推送凭据；beta 默认只定时生成本地元数据归档。
- 明确源码安装是首发分发方式；当前不承诺独立 PyPI wheel 能脱离仓库资源运行。

## 发版门槛

本机跨 Agent 安装验收、控制面契约、插件校验和 GitHub Actions 通过后，可以发布 `v0.2.0-beta.5`。干净电脑的真实安装、微信扫码或 RSS 模式、回补、跨任务恢复、幂等恢复、结果验收与报告读取通过后，再发布 `v0.2.0` 稳定版。
